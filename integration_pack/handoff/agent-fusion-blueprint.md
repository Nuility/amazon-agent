# 后续融合到 Agent 项目的实施蓝图

## 目标

把 Wimoor 的亚马逊广告投放能力封装成 agent 可调用工具，并用 CopilotKit 风格的人机协同 UI 承接用户的自然语言分析、建议生成和受控执行。

## 建议实施顺序

1. **建立广告工具层**
   - 按 `contracts/advertising-agent-contract.yaml` 实现只读工具。
   - 第一批只接报表、活动列表、账户概览，不做写操作。

2. **接入 Wimoor 广告后端**
   - 若目标项目允许 Java 服务并存，保留 `wimoor-amazon-adv` 为独立服务。
   - 若目标项目以 Python agent 为主，则用 HTTP/API 方式调用 Wimoor 广告服务，不直接翻译全部 Java 业务代码。

3. **加入分析与建议**
   - 把 ACOS、ROAS、CTR、CPC、CVR 等指标标准化。
   - 先输出可解释建议，不直接执行。

4. **加入 CopilotKit 交互层**
   - 在广告页面放入 Copilot 入口。
   - 将当前店铺、站点、profile、时间范围同步给 agent。
   - 将建议以可确认的 UI 事件呈现。

5. **开放受控写操作**
   - 预算、竞价、暂停、关键词新增等动作必须确认。
   - 所有写操作生成审计日志和回滚提示。

## 验收标准

- 用户能问：“最近 7 天哪个广告活动 ACOS 异常升高？”
- Agent 能拉取报表、给出原因解释和优化建议。
- 用户能看到建议涉及的广告对象、旧值、新值和风险等级。
- 高风险动作必须确认后才执行。
- 后续可把工具层从模拟数据切换到 Wimoor 广告 API，而不改变 UI 交互契约。

## 默认技术选择

- Wimoor 广告模块保留为外部服务或源码参考，不直接混进 agent 核心目录。
- Agent 工具契约使用 YAML 先描述，后续再生成 Python/TypeScript 类型。
- CopilotKit 作为前端依赖引入，不复制 monorepo。
- 第一阶段只读，第二阶段建议，第三阶段受控执行。
